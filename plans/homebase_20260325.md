● Given the goal (understand → audit → architecture decision → revise ADR),
  this is a three-phase research-then-design workflow, not a
  planner/implementation task. Here's the exact sequence:

  ---
  Phase 1 — Codebase Orientation

  /codebase-analysis

  Run this first with no prior exploration. It orchestrates hierarchical codebase discovery and returns a structured map of what exists, how it's layered, and where the boundaries are. Feed it a scope hint:

  ▎ "Analyze the full codebase with focus on the hexagonal architecture in src/core/ + src/infrastructure/, the Next.js API layer, and the Expo mobile app in apps/mobile/. Identify layering, coupling points, and what's actually using the hexagonal scaffold vs bypassing it."

  Output to save: The analysis report. You'll use it as input to Phase 2.

  ---
  Phase 2 — Issue Audit + Architecture Stress-Test

  Run these sequentially (second feeds the third):

  2a. /problem-analysis

  ▎ "Given this codebase analysis [paste output], identify current architectural issues: which parts of the hexagonal scaffold are actually exercised, where API routes bypass the ports/adapters pattern, and what friction points exist for the React Native mobile app consuming the same backend."

  This surfaces concrete problems — not hypothetical ones.

  2b. /decision-critic

  ▎ "Stress-test this architectural decision: hexagonal architecture (ports + adapters) vs Vertical Slice Architecture for a system that is simultaneously a Next.js web app AND an Expo React Native companion app sharing the same API backend. Current state: hexagonal scaffold exists in src/core/ but most API routes live outside it. Mobile app is Phase M1. The system will grow to ~13 phases. What are the failure modes of each approach at this scale and team size?"

  The critic will challenge assumptions and surface the real tradeoffs — particularly whether hexagonal makes sense when the primary consumer is now a mobile app where feature cohesion (VSA) is the dominant RN community pattern.

  ---
  Phase 3 — ADR Revision

  Invoke the architect agent directly (not /planner — this is design, not implementation):

  ▎ "Using these findings [paste problem-analysis + decision-critic outputs], revise docs/ADR.md. The ADR should document: (1) the original hexagonal decision and its rationale, (2) the evaluated alternative of VSA, (3) the recommended approach for the mobile layer specifically vs the web layer, (4) the migration path if the recommendation diverges from the current scaffold, (5) constraints that must not change (org_id isolation, JARVIS hexagonal scaffold for Phases 12-13). Follow .claude/conventions/documentation.md — ADR captures invisible knowledge."

  Then run technical-writer to finalize formatting, and optionally quality-reviewer to check the ADR for DECISION_LOG_MISSING or ASSUMPTION_UNVALIDATED findings.

  ---
  Execution Summary

  /codebase-analysis  →  /problem-analysis  →  /decision-critic  →  architect (ADR revision)  →  technical-writer

  Why not /planner here? Planner orchestrates implementation milestones with developer delegation. This workflow is pure research + design — no code changes until Phase 4 executes the ADR decision.

  Context hygiene: After each phase, paste the output into the next prompt explicitly. The skills start fresh — they don't share memory between invocations.

  One gotcha for this specific codebase: The hexagonal scaffold (src/core/) was designed for JARVIS (Phases 12-13). VSA might be right for mobile while hexagonal stays correct for the eventual self-hosted backend. The decision-critic should be pushed on whether these can coexist rather than forcing a binary choice.
