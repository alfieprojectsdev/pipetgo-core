  ---
  Session 1 — Explore + Plan (60 min)

  Goal: Understand every personal data collection point, resolve the consent persistence
  question, produce the plan file.

  Step 1 — Branch (skip; plan is written on main, branch created at execution time)

  Step 2 — Explore (run before calling the planner)

  Say:
  Read prisma/schema.prisma (ClientProfile model and ServiceCategory enum), then read
  src/features/orders/create-order/action.ts, src/features/orders/create-order/ui.tsx,
  and src/domain/orders/client-details.ts. Summarise: (1) every field currently written
  to ClientProfile, (2) how clientDetailsSchema is used as the sole validator in the
  action, (3) whether any consent fields already exist on ClientProfile or in the schema.
  Then read src/features/orders/create-order/README.md if present.

  Step 3 — Write the plan

  Use your planner skill to write a plan to plans/T-20-privacy-compliance.md

  The planner runs the interview → steelman → TW + QR gate cycle. Key decisions the
  plan must resolve:

  - Consent persistence: ClientProfile needs two new fields — consentGiven Boolean and
    consentGivenAt DateTime? — so RA 10173 consent can be demonstrated per-order. The
    plan must include a Prisma schema migration for these. consentGiven must be validated
    as z.literal(true) in clientDetailsSchema (not just z.boolean()) — an unchecked box
    is a hard submission block, not a soft warning.

  - Sensitive data flag: CHEMICAL_TESTING and BIOLOGICAL_TESTING are already defined in
    ServiceCategory (prisma/schema.prisma line ~65). No new DB column is needed — the
    plan must document that these two categories ARE the sensitive-data flag, and that
    the retention policy note belongs in a README / admin runbook, not a new column.

  - Privacy notice page: /src/app/privacy/page.tsx — static RSC, no auth required.
    Content must cover: data controller identity (PipetGo / Alfie), purpose of
    collection, categories of data collected (contact details, test order details),
    retention period, NPC contact, and data subject rights (access, rectification,
    erasure). Plan must include a note that legal review is required before production
    use; stub copy is acceptable for the PR.

  - Data subject deletion flow: email-based only at this stage. The plan must specify
    the admin runbook (mailto: link + documented manual process) rather than a
    self-service UI. A self-service deletion slice is out of scope for T-20.

  - Consent checkbox placement: inline in the create-order form, after the contact
    fields, before the submit button. Must be a required field (cannot submit without
    checking). Must link to /privacy in a new tab.

  - clientDetailsSchema must remain the sole validator — do not add a separate Zod
    schema in the action or UI.
