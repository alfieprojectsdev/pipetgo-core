       Name: pipetgo-kyc-docs (or similar)
       Location: Asia Pacific (apac) — closest to PH
       Default encryption: on

    2. Create an R2 API token (Settings → R2 API Tokens → Create token)
       Permissions: Object Read & Write
       Scope: specific bucket → pipetgo-kyc-docs
       Save: CLOUDFLARE_ACCOUNT_ID, R2_ACCESS_KEY_ID, R2_SECRET_ACCESS_KEY

    3. Add to local .env.local (not committed):
       CLOUDFLARE_ACCOUNT_ID=...
       R2_ACCESS_KEY_ID=...
       R2_SECRET_ACCESS_KEY=...
       R2_BUCKET_NAME=pipetgo-kyc-docs
       R2_ENDPOINT=https://<CLOUDFLARE_ACCOUNT_ID>.r2.cloudflarestorage.com

    4. Note for Vercel: these five vars must be added to Vercel env before deploy.
       Add them to the DevOps provisioning checklist in docs/roadmap.md.

  Free plan limits (confirm before committing to R2):
    - 10 GB-months storage
    - 1M Class A ops/month (PUT, POST, LIST)
    - 10M Class B ops/month (GET, HEAD)
    - Zero egress fees (unlike S3)
  Sufficient for MVP KYC doc volume.

  Step 2 — Explore (run before calling the planner)

  Say:
  Read prisma/schema.prisma focusing on: (1) the Lab model (isVerified field and
  its current meaning), (2) the Attachment model — note that orderId is String
  (NOT NULL), meaning KYC docs tied only to a Lab (no Order) cannot use the
  current Attachment model without a schema change, (3) AttachmentType enum
  (current values: SPECIFICATION, RESULT, ACCREDITATION_CERTIFICATE). Then read
  src/features/payments/payouts/handlers.ts (T-10 settlement handler) to
  understand where a KYC gate would be inserted. Then read
  src/lib/payments/xendit.ts to see what Xendit API calls already exist.

  The planner must resolve these before writing the plan:
    - Attachment.orderId is currently required — KYC docs are lab-level, not
      order-level. Plan must choose: (a) make orderId optional (String?) via
      migration, or (b) add a new LabDocument model for lab-level files. Do not
      silently reuse ACCREDITATION_CERTIFICATE with a null orderId — that breaks
      the existing FK constraint.
    - Lab.isVerified is reserved for T-18 ISO 17025 accreditation. T-15 needs a
      separate Lab.kycStatus field (enum: PENDING, SUBMITTED, APPROVED, REJECTED)
      for Xendit payment gateway KYC. These are different regulatory concerns.
    - Xendit KYC API: research the business verification endpoint. Key questions:
      which endpoint accepts documents, what is the payload shape (file URL vs
      base64 vs multipart), does Xendit poll or fire a webhook for status updates,
      and does the existing verifyXenditToken cover the KYC webhook or is a
      separate token needed?

  Step 3 — Write the plan

  Use your planner skill to write a plan to plans/T-15-lab-kyc-upload.md
arp: ```
claude --resume a481d686-df1c-400b-87e5-21fc304ee68d
```
  The planner runs the interview → steelman → TW + QR gate cycle. Key decisions
  the plan must resolve:

  - Schema migration: two changes needed:
      (a) Lab.kycStatus KycStatus enum field (PENDING, SUBMITTED, APPROVED,
          REJECTED) @default(PENDING) — separate from isVerified
      (b) Attachment.orderId: make optional (String?) OR introduce LabDocument
          model. Document which is chosen and why (orderId optional is lower
          surface; LabDocument is cleaner separation). This affects whether the
          existing Attachment model is reused or bypassed.
      (c) New AttachmentType members if needed: KYC_BIR_2303, KYC_DTI_SEC — or
          document why ACCREDITATION_CERTIFICATE is reused.

  - R2 upload pattern: presigned PUT URL (preferred — avoids Next.js 4.5 MB
    FormData limit for a 20 MB file ceiling). Flow:
      1. LAB_ADMIN selects file in browser
      2. Client component calls a Server Action to generate a presigned PUT URL
         (signed by server using R2 credentials — key: labs/{labId}/{uuid}.{ext})
      3. Client uploads directly to R2 via fetch PUT to the presigned URL
      4. On upload success, client calls a second Server Action to record the
         Attachment (or LabDocument) in DB and submit the URL to Xendit KYC API
    Server-side direct upload is an alternative but requires streaming the full
    file through Next.js — not suitable for 20 MB docs on Vercel's 4.5 MB limit.

  - src/lib/storage/r2.ts: new file. Exports:
      generatePresignedPutUrl(key: string, mimeType: string, maxBytes: number): Promise<string>
      generatePresignedGetUrl(key: string): Promise<string>
    Uses @aws-sdk/client-s3 + @aws-sdk/s3-request-presigner (S3-compatible R2
    endpoint). Put URL TTL: 300 s. Get URL TTL: 3600 s.
    Env vars: R2_ENDPOINT, R2_ACCESS_KEY_ID, R2_SECRET_ACCESS_KEY, R2_BUCKET_NAME.

  - Xendit KYC submission: src/lib/payments/xendit.ts — add submitLabKyc(). Plan
    must document the exact Xendit endpoint, required fields, and whether the
    function is called synchronously in the Server Action (after DB record) or
    queued. If Xendit returns a reference ID, store it on Lab or LabDocument.

  - KYC status webhook: if Xendit fires a webhook on KYC status change, add
    src/app/api/webhooks/xendit-kyc/route.ts (follows T-14 pattern: verify token →
    update Lab.kycStatus). If Xendit only supports polling, document the polling
    approach and whether a cron or manual admin action drives it.

  - Payout gate: the T-10 settlement handler (processSettlement) currently has no
    KYC check. The plan must specify the gate: before creating a commission Payout
    record or before any Xendit sub-account interaction, confirm Lab.kycStatus ===
    APPROVED. If not APPROVED, throw (not silently skip) — same pattern as the
    LabWallet missing guard in handlers.ts.

  - UI: new slice src/features/labs/kyc/ with:
      page.tsx — RSC, shows current kycStatus badge + list of uploaded docs
      upload-action.ts — generates presigned URL + records Attachment/LabDocument
      confirm-action.ts — receives upload confirmation, calls Xendit KYC submit,
      updates Lab.kycStatus = SUBMITTED
    Reachable from lab dashboard nav (existing /dashboard/lab layout).

  - File validation: enforced server-side at presigned URL generation.
      Allowed MIME types: application/pdf, image/jpeg, image/png
      Max size: 20 MB (20 * 1024 * 1024 bytes)
    Plan must document where the size check happens — at presigned URL generation
    (pass Content-Length condition to presigned PUT) or at confirm-action (read
    fileSize from FormData metadata). Both; belt-and-suspenders.
