# T-18 — Lab accreditation verification (ISO 17025 / ITA solidary liability)

Plan document: `plans/T-18-lab-verification.md` (to be written)
Branch: `feat/T18-lab-verification`
Estimated sessions: 2 | Estimated wall time: ~3 hours
Why this is next: T-13 merged (PR #17) was T-18's only remaining blocker. Under the
Internet Transactions Act (ITA) of 2023, PipetGo has **solidary liability** for unsafe or
unaccredited services it lists. `Lab.isVerified` exists in the schema but is enforced
nowhere — today any KYC-approved lab's services appear in the marketplace regardless of
DTI-PAB ISO 17025 accreditation. T-18 adds the accreditation review surface (admin) and the
two enforcement gates (marketplace + create-order).

T-18 is **substantially de-risked by T-13 + T-15** — the admin auth surface, the
review-slice shape, `generatePresignedGetUrl`, and the lab-scoped `LabDocument` model all
already exist. This is mostly *cloning* established patterns + adding two read gates, hence
the 2-session estimate (vs T-13's 3).

---

## Pre-session — resolve these BEFORE calling the planner

Three decisions must be nailed before the plan is written. Do not let the planner re-open
them mid-interview.

### Decision 1 (the big one): storage model — reuse `LabDocument`, NOT `Attachment`

The roadmap ticket says the cert is "stored as `AttachmentType.ACCREDITATION_CERTIFICATE`".
**That wording predates T-15.** The `Attachment` model is **order-scoped** (`Attachment.orderId`
is NOT NULL — this is exactly why T-15 created `LabDocument` instead of reusing it). An ISO
17025 accreditation certificate is a **lab-level** document, not tied to any order.

**DECIDED: reuse `LabDocument`** with `documentType = 'ACCREDITATION_CERTIFICATE'`.
`LabDocument` already has: `labId` scope, `r2Key @unique`, `status DocumentStatus`
(PENDING/UPLOADED/VERIFIED/REJECTED), R2 presigned PUT (upload) **and** GET (admin view, added
in T-13). This means **zero new storage infra**. The `AttachmentType.ACCREDITATION_CERTIFICATE`
enum value becomes dead — note it in the plan; do not wire it.
- Rejected alt: make `Attachment.orderId` nullable — a far larger, riskier migration that
  touches every existing `Attachment` query's null-assumptions. Avoid.

### Decision 2: `isVerified` is independent of `kycStatus` — do not conflate

Two separate lab gates, two separate lifecycles (already documented in
`src/features/admin/kyc-review/README.md`):
- `kycStatus` (T-15/T-13) — **payment gate**. APPROVED ⇒ lab can take payment at checkout.
- `isVerified` (T-18, this ticket) — **marketplace-visibility / ITA accreditation gate**.
  true ⇒ lab's services are listed and orderable.

A lab can be `kycStatus = APPROVED` (can receive money) yet `isVerified = false` (hidden from
marketplace). T-18 touches **only** `isVerified`; it must not read or write `kycStatus`.

### Decision 3: marketplace gate is a BEHAVIOR CHANGE — plan the rollout

Today `/services` lists every active service. Gating on `lab.isVerified === true` will
**empty the marketplace** until labs are verified (no lab has `isVerified = true` yet).
Decide the rollout:
- Verify the founder/seed lab first (manual `UPDATE labs SET "isVerified" = true …`, recorded
  in the DevOps checklist), OR
- Ship the gate but communicate that the marketplace is intentionally empty until the first
  lab is accredited.
Either is fine — but the plan must state it so the empty `/services` isn't read as a bug.

**Audit fields** (resolve in plan): mirror the T-13 KYC audit columns —
`accreditationReviewedById`, `accreditationReviewedAt`, `accreditationRejectionReason` on
`Lab` (a rejection reason shown back to the lab is a real UX need). This is a schema change;
migrations are gitignored — apply locally (see gotcha #6).

The planner must resolve before writing:

- **Clone vs extend the admin slice.** Recommended: a **new** sibling slice
  `src/features/admin/accreditation-review/` cloning the kyc-review shape (queue + detail +
  approve/reject action + view-document-action). VSA (ADR-001) forbids cross-slice imports —
  badge maps and DTO types are **copied**, not imported from kyc-review. The admin
  `layout.tsx` guard is shared infra (route group) and is reused as-is.

- **The review queue filter.** kyc-review filters `kycStatus = SUBMITTED`. Accreditation has
  no equivalent status on `Lab`. Filter the queue on labs that have at least one
  `LabDocument` with `documentType = 'ACCREDITATION_CERTIFICATE'` and `status = UPLOADED`,
  and `isVerified = false`. Decide the exact `findMany` + `where` shape (e.g. `some`
  relation filter) and oldest-first ordering.

- **The verify transition is a boolean CAS.** Setting `isVerified false → true` must use
  `updateMany({ where: { id, isVerified: false }, data: { isVerified: true, … } })` +
  `count === 0` early-return — never a bare `update` (concurrent-admin guard, mirrors T-13
  DL-002). Reject path: keep `isVerified = false`, record the rejection reason, cascade the
  cert `LabDocument` `UPLOADED → REJECTED`; approve path cascades `UPLOADED → VERIFIED`. Same
  `$transaction` (mirrors T-13 DL-007).

- **Lab-side cert upload.** Clone/extend the kyc-upload slice for a `LAB_ADMIN` to upload the
  ISO 17025 cert via presigned PUT with `documentType = 'ACCREDITATION_CERTIFICATE'`. Decide:
  a new slice `src/features/labs/accreditation-upload/` vs parameterising kyc-upload by
  documentType. A separate slice is cleaner (different copy, different gate) and lower-risk.
  Surface the lab's accreditation status (and any rejection reason) back on the lab dashboard.

- **Marketplace gate (read).** `src/features/services/browse/page.tsx` query must filter to
  services whose `lab.isVerified === true`. Confirm the relation filter and that the empty-
  marketplace rollout (Decision 3) is documented.

- **create-order gate (read, security-critical).** `src/features/orders/create-order/action.ts`
  must reject order creation when the target service's `lab.isVerified === false` —
  **server-side**, not just UI hiding. A client can POST a serviceId directly; UI hiding is
  not a security control. This is the ITA-liability-bearing guard.

- **Audit fields + migration** (Decision in Pre-session). New `Lab` columns; apply locally.

## Critical gotchas (carry into the plan)

1. **Reuse `LabDocument`, not `Attachment`.** The cert is lab-scoped; `Attachment.orderId` is
   NOT NULL (order-scoped — the reason T-15 made `LabDocument`). Use
   `LabDocument` + `documentType = 'ACCREDITATION_CERTIFICATE'`. The
   `AttachmentType.ACCREDITATION_CERTIFICATE` enum value stays dead — do not wire it.
2. **`isVerified` ≠ `kycStatus`.** Independent gates, independent lifecycles. T-18 reads/writes
   only `isVerified`; it must not touch `kycStatus`. (already documented in kyc-review/README.md)
3. **Admin infra already exists** (T-13). Reuse `src/app/dashboard/admin/layout.tsx` and
   `generatePresignedGetUrl`. Clone `src/features/admin/kyc-review/` → `accreditation-review/`;
   **copy** badge maps/DTOs (VSA — no cross-slice imports, ADR-001).
4. **Verify is a boolean CAS.** `updateMany({ where: { id, isVerified: false } })` +
   `count === 0` early-return — never a bare `update`. Cascade the cert `LabDocument`
   `UPLOADED → VERIFIED|REJECTED` in the same `$transaction`.
5. **The create-order gate is the security-critical one.** Hiding services in the marketplace
   UI is not a control — a client can POST a `serviceId` directly. The server-side reject in
   `create-order/action.ts` is the ITA-liability-bearing guard. Both gates ship together.
6. **Migrations are gitignored; dev DB is a `db push` workflow.** Apply the audit-column
   change with `npx prisma db push` (NOT `migrate dev` — it would drift/reset against the
   push-managed dev DB; see `docs/sessions/2026-05-30-T13-implementation-done.md`). Do not
   commit migration files. `schema.prisma` is the committed source of truth.
7. **Empty marketplace is expected** post-gate until the first lab is verified — bootstrap the
   founder lab via documented manual `UPDATE` or verify it through the new UI.

## Implementation Discipline reminders (from CLAUDE.md)

- Admin action TOCTOU: re-check `role === 'ADMIN'` inside every accreditation Server Action —
  the layout guard does not protect actions.
- `updateMany` + guard predicate (`isVerified: false`) + `count === 0` early-return for the
  verify write; unhandled/unexpected state throws.
- `findUnique` on `@unique` fields (`Lab.id`, `Lab.ownerId`, `LabDocument.id`) — never `findFirst`.
- Null relation after explicit `include` → throw; missing row / wrong owner → `notFound()`.
- RSC DTOs: `Decimal` → `.toFixed(2)`, `Date` → `.toISOString()` before crossing to a client component.
- `redirect()` after — never inside — try/catch in Server Actions.
- Enum/badge maps via `as const satisfies Record<DocumentStatus,…>` — copied into the slice, not imported.
- Presigned GET URL key is server-trusted (`LabDocument.r2Key`), never client-supplied; 300 s TTL.
- Boundary inputs runtime-narrowed (`typeof x === 'string'`), no masking `?? ''` / `?? null` on
  contract-guaranteed values (newest Compounding Protocol bullet, PR #17).
