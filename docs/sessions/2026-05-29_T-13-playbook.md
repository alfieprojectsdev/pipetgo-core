# T-13 — Admin panel (lab KYC approval + admin auth surface)

Plan document: `plans/T-13-admin-panel.md`
Branch: `feat/T13-admin`
Estimated sessions: 3 | Estimated wall time: ~4.5 hours
Why this is the priority: T-15 merged the KYC gate on **both** checkout paths
(`initiateCheckout` + `initiateVaCheckout`). Labs now default to `kycStatus = PENDING`
and **cannot reach `APPROVED` without an admin UI**. Until T-13 ships, a direct DB

## Pre-session — resolve scope BEFORE calling the planner

T-13's scope is deliberately undefined in the roadmap ("plan must define the surface
area before implementation"). The roadmap lists three candidate surfaces:

1. **Lab KYC review + approval** — set `Lab.kycStatus = APPROVED | REJECTED` after viewing uploaded `LabDocument`s
2. **User role management** — promote/demote `UserRole` (CLIENT ↔ LAB_ADMIN ↔ ADMIN)
3. **Order oversight** — admin read view of all orders / transactions

**DECIDED (2026-05-29): first PR ships surface #1 only** — KYC review/approve plus the
admin auth infrastructure (route group + role gate + first-admin bootstrap). Surfaces #2
(role management) and #3 (order oversight) are spun out as T-13b; `isVerified` stays with
T-18. The planner should not re-open this — proceed with #1.

Reasons:
- #1 is the one blocker preventing first revenue — it is what "priority↑" refers to.
- #2 (role management) is a privilege-escalation surface; it deserves its own plan, audit trail, and review pass — do not bundle it under time pressure.
- #3 (order oversight) is read-only and low-risk — fold it in only if Session 2 finishes early; otherwise spin it out as T-13b.
- `Lab.isVerified` (ISO 17025) is **T-18's** enforcement field. T-13 may add the *write surface* for it on the same admin lab-detail page, but the marketplace gate that *reads* it is T-18. Decide explicitly: write-now-gate-later, or leave `isVerified` untouched until T-18.

If the user wants the broader panel in one PR, budget +1 session per extra surface.

**First-admin bootstrap** (must be resolved in the plan — there is no ADMIN user today):
- Option A (recommended): a one-off `prisma/seed` or documented manual `UPDATE users SET role='ADMIN'` for the founder account, recorded in the DevOps checklist. Lowest surface.
- Option B: env allowlist (`ADMIN_EMAILS`) checked in the `jwt` callback in `src/lib/auth.ts`. More moving parts; only if self-service admin promotion is wanted.
Do **not** build a UI that lets an ADMIN mint the first ADMIN — chicken-and-egg.

### Step 2 — Explore (run before calling the planner)

Say:
> Read `src/lib/auth.ts` (JWT role callback, no middleware in this repo), then read how
> existing slices gate by role: `src/features/orders/lab-fulfillment/page.tsx` and
> `action.ts` (`session.user.role !== 'LAB_ADMIN'` per-page + per-action TOCTOU re-check).
> Then read the T-15 KYC data model: `prisma/schema.prisma` (`KycStatus`,
> `DocumentStatus`, `Lab.kycStatus`, `Lab.isVerified`, `LabDocument`), and the T-15 slice
> `src/features/labs/kyc-upload/` (`confirm-action.ts` `updateMany` CAS pattern, `ui.tsx`
> `as const satisfies Record<KycStatus,…>` badge maps). Finally read
> `src/lib/storage/r2.ts` and confirm it exports **only** `generatePresignedPutUrl` — there
> is no GET-URL function, so the admin cannot view an uploaded document yet.

The planner must resolve these before writing the plan:

- **Admin auth surface.** There is no `middleware.ts` and no `/dashboard` layout in this
  repo — gating is per-page/per-action. Decide: add a route-group layout guard at
  `src/app/dashboard/admin/layout.tsx` (RSC: `auth()` → `role !== 'ADMIN'` → `notFound()`
  or `redirect('/auth/signin')`) **and** re-check role inside every admin Server Action
  (TOCTOU — a layout guard does not protect actions). Document that both layers are
  required, matching the lab-fulfillment precedent (DL-001).

- **Document viewing — the r2.ts gap.** `r2.ts` has no `generatePresignedGetUrl`. The
  admin must view BIR 2303 / DTI-SEC PDFs to make an approval decision. The plan must add
  `generatePresignedGetUrl(key: string): Promise<string>` to `src/lib/storage/r2.ts`
  (TTL ~300 s, behind `server-only`, `AbortSignal`/`requestTimeout` already on the client).
  Decide whether the admin page server-generates a short-lived GET URL per document on
  render, or an action mints one on click. Server-on-render is simpler; note the URL must
  not be cached in the RSC payload longer than its TTL.

- **Approval = a guarded state transition, not a bare update.** Setting
  `Lab.kycStatus PENDING|SUBMITTED → APPROVED|REJECTED` is a status transition. Per
  Implementation Discipline it must use `updateMany({ where: { id, kycStatus: <expected> }, … })`
  + `count === 0` early-return, **not** a bare `update`. Decide the allowed source states
  (likely `SUBMITTED → APPROVED|REJECTED`; reject approving a `PENDING` lab that never
  uploaded). Unhandled source state must throw.

- **Cascade to LabDocument status.** When a lab is approved/rejected, do the
  `LabDocument.status` rows move `UPLOADED → VERIFIED | REJECTED` in the same
  `$transaction`? Decide and make it atomic.

- **Audit trail.** Who approved, when, and why (rejection reason)? Decide: new fields on
  `Lab` (`kycReviewedById`, `kycReviewedAt`, `kycRejectionReason`) or a dedicated
  `AdminAuditLog` model. A rejection reason shown back to the lab is a real UX need —
  prefer at least the fields. This is a schema migration (gitignored migrations — apply
  locally; see note at end).

- **Lab list + filter.** The admin landing needs a list of labs filterable by
  `kycStatus` (default to `SUBMITTED` — the review queue). Confirm `findMany` ordering
  (oldest submission first) and that all `Decimal`/`Date` fields are serialized
  (`.toFixed(2)` / `.toISOString()`) before crossing the RSC boundary.

- **`isVerified` scope decision** (see Pre-session) — in or out of this PR.

## Critical gotchas (carry into the plan)

1. **No middleware, no dashboard layout.** Gating is per-page/per-action. The admin route
   group needs its own `layout.tsx` guard **and** per-action role re-checks. (precedent:
   `src/features/orders/lab-fulfillment/` — DL-001, `LAB_ADMIN` per-page + per-action)
2. **`r2.ts` has no GET URL.** Only `generatePresignedPutUrl` exists. Admin doc review is
   impossible until `generatePresignedGetUrl` is added. This is the single most likely
   thing to be missed — it is a hard dependency of the review UI.
3. **No ADMIN user exists.** `UserRole.ADMIN` is in the enum but unused anywhere. Bootstrap
   the first admin out-of-band; do not build self-service admin minting.
4. **Two separate lab gates:** `kycStatus` (T-15, payment gate, this ticket) vs
   `isVerified` (T-18, ISO 17025 marketplace gate). Do not conflate. KYC ≠ accreditation.
5. **Approval is a state transition** — `updateMany` CAS + `count === 0`, throw on
   unhandled source state. Reuse the exact pattern from
   `src/features/labs/kyc-upload/confirm-action.ts`.
6. **Migrations are gitignored** in this repo — apply locally (`npx prisma migrate dev …`);
   do not commit migration files in the PR.

## Implementation Discipline reminders (from CLAUDE.md)

- Admin action TOCTOU: re-check `role === 'ADMIN'` inside every Server Action — layout
  guard alone is insufficient.
- `findUnique` on `@unique` (`Lab.ownerId`, `Lab.id`) — never `findFirst`.
- `updateMany` + guard predicate + `count === 0` early-return for the `kycStatus` write.
- Enum dispatch via `as const satisfies Record<KycStatus|DocumentStatus,…>`.
- Null relation after explicit `include` → throw; missing row / wrong owner → `notFound()`.
- RSC DTOs: `Decimal` → `.toFixed(2)`, `Date` → `.toISOString()`.
- `redirect()` after — never inside — try/catch.
- Presigned GET URL key is server-trusted (`LabDocument.r2Key`), never client-supplied.
