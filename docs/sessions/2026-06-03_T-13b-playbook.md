# T-13b — Admin order oversight (read-only)

Plan document: `plans/T-13b-admin-order-oversight.md` (to be written)
Branch: `feat/T13b-admin-order-oversight`
Estimated sessions: 1–2 | Estimated wall time: ~1.5 hours
Why this is next: the only `[ready] [planner]` ticket left. T-13c (admin role management) is
deferred behind its own privilege-escalation audit; T-19 (dispute/redress) is blocked on a
schema migration. T-13b is **read-only** — the lowest-risk admin surface — and reuses the
admin layout guard + two-layer auth pattern already shipped in T-13. It gives an ADMIN a
single place to see every order, its payment transaction(s), and its payout(s) across the
platform, which is the operational visibility a launched marketplace needs before first
commercial transaction.

T-13b is **substantially de-risked by T-13 + T-18**: the `/dashboard/admin/*` route group,
its layout guard (`src/app/dashboard/admin/layout.tsx`), the ADMIN re-check pattern in Server
Actions, and the list-page + `[id]`/detail-page shape (`src/features/admin/kyc-review/`) all
exist. This is mostly *applying* that established admin slice pattern to read-only
order/transaction/payout data — hence the 1–2 session estimate.

**Key difference from T-13/T-18:** those wrote a mutating action (approve/reject, verify/reject)
guarded by a status transition. **T-13b writes NOTHING.** No Server Action mutates state; there
is no schema change and therefore **no `npx prisma db push` owed** (unlike T-12/T-18 — call this
out so no one burns a session looking for a migration that does not exist).

## Pre-session — resolve these BEFORE calling the planner

Front-load these; do not let the planner re-open them mid-interview.

### Decision 1 (the boundary): T-13b is STRICTLY READ-ONLY — no mutation, ever

The tempting scope creep is a "refund this order" / "force status" / "reassign lab" button.
**All of those are out.** Any write to an order, transaction, or payout from an admin surface
is a separate, higher-risk ticket (it touches money movement and the payment-event state
machine). If the planner or a reviewer proposes a write action, the answer is: **defer to a
dedicated ticket.** T-13b ships read views only — `findMany` lists + `findUnique` detail, no
`update`/`updateMany`/`create`/`delete` anywhere in the slice.

Role grant/revoke (`UserRole`) is **T-13c**, explicitly deferred. Do not wire it here.

### Decision 2: two-layer auth applies even to read-only pages (DL-001 TOCTOU)

The layout guard (`src/app/dashboard/admin/layout.tsx`) is **layer 1** only — it redirects
non-ADMIN sessions before rendering children, but it does **not** protect Server Actions,
which are independently POST-invocable. Every Server Action in this slice (e.g. a presigned-GET
`viewOrderAttachment`-style action, or any data-fetch action invoked from a client component)
**must independently re-check `session.user.role === 'ADMIN'`** — exactly as `kyc-review` does.
RSC page components that read admin data must also re-`auth()` and re-check ADMIN at the top,
not rely on the layout alone. This is admin-only data exposure; the cost of getting the gate
wrong is a cross-tenant data leak.

### Decision 3: RSC serialization is mandatory — orders/transactions/payouts are Decimal- and Date-heavy

Per the standing CLAUDE.md rule, every `Prisma.Decimal` must be serialized via `.toFixed(2)`
and every `Date` via `.toISOString()` **before** the data crosses the RSC → client-component
boundary, and the DTO field types must reflect the serialized form (`amount: string`, not
`Decimal`). The fields in scope:

- `Order.quotedPrice` (`Decimal?`), `quotedAt` / `paidAt` / `refundedAt` / `createdAt` / `updatedAt` (Dates)
- `Transaction.amount` (`Decimal`), `capturedAt` / `refundedAt` / `createdAt` (Dates)
- `Payout.grossAmount` / `platformFee` / `netAmount` / `feePercentage` (`Decimal`), `scheduledDate` / `completedAt` (Dates)

A raw Decimal/Date across the boundary is a **runtime crash, not a type error** — it will not
show up in `tsc`. Define explicit DTOs (`AdminOrderRowDTO`, `AdminOrderDetailDTO`, …) with
string-typed money/date fields and map at the page boundary.

### Decision 4: RA 10173 / PII minimization — decide what the oversight view surfaces

An order joins `ClientProfile` (name, email, phone, organization, address — all PII) and the
client `User`. Admin oversight legitimately needs *some* of this for support, but the Data
Privacy Act favors data minimization. **DECIDE up front** which fields the list vs the detail
view show:

- **(Recommended)** List view: no raw PII — order id, status, lab name, amount, timestamps,
  client *display name* only. Detail view: full ClientProfile, justified by the support use
  case, behind the ADMIN gate.
- Surface full PII everywhere (simpler, but weaker minimization posture).

Whichever is chosen, **state it in the plan** and add a one-line note that this view processes
PII (relevant when the RA 10173 retention/deletion process is defined before first commercial
transaction — see roadmap T-12 retention flag). Do **not** add analytics/event tracking to
these pages (global standard: never track personal or financial data).

### Decision 5: list shape — filter by status, order by recency, read ALL rows

Unlike client/lab pages, admin oversight is **unscoped** — no `clientId`/`labId` predicate;
the admin sees every order. Mirror `kyc-review`'s list+detail layout:

- `src/app/dashboard/admin/orders/page.tsx` → list (filter by `OrderStatus`, default order
  `createdAt desc`, paginate or cap rows).
- `src/app/dashboard/admin/orders/[orderId]/page.tsx` → detail (order + its transactions +
  payouts + attachments metadata).

Use `findUnique` for the detail lookup on the `@id`; after an explicit `include`, a null
relation that the schema guarantees (e.g. `order.lab`) must `throw`, not `notFound()` — only a
genuinely missing order is `notFound()` (DL bullet, see `quote-provide/page.tsx`).

## Watch-points during implementation

- **No write paths.** Grep the finished slice for `update`/`create`/`delete`/`upsert` — there
  should be none. If one appears, it is out of scope (Decision 1).
- **Layer-2 ADMIN re-check** in every action and every RSC admin page (Decision 2).
- **Decimal/Date serialization** at the boundary (Decision 3) — the recurring compounding-rule
  failure; type the DTOs as `string`.
- **`findMany` for lists, `findUnique` for the `@id` detail** — never `findFirst`.
- **Unscoped reads are intentional** but must stay ADMIN-gated; double-check no admin query
  leaks into a non-admin page.

## Tests (vitest.unit.config.ts — no-DB full-mock)

- Non-ADMIN session is rejected by each Server Action (layer-2 gate), not just the layout.
- DTO mapping serializes every Decimal (`toFixed(2)`) and Date (`toISOString`) — assert string
  shape, not Decimal/Date.
- List ordering (`createdAt desc`) and status filter pass the predicate through to `findMany`.
- Detail throws on a null guaranteed relation after include; `notFound()` only on missing order.
- Register the new `__tests__` globs in `vitest.unit.config.ts`.
