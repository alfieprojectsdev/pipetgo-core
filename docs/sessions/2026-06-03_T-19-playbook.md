## Pre-session — resolve these BEFORE calling the planner

### Decision 1 (the schema change): add `DISPUTED` to `OrderStatus` — migration + `db push` owed

One enum value: `DISPUTED`. This is additive (safe), but `prisma/migrations/` is gitignored
(DL-011) — `schema.prisma` is the committed source of truth, and **every environment owes
`npx prisma db push` after pulling** (dev/CI/prod) or the new status crashes at runtime, not at
`tsc`.

**The badge-map trap (verified 2026-06-03):** only **two** files use
`as const satisfies Record<OrderStatus,…>` — `src/features/admin/order-oversight/ui.tsx` and
`detail-ui.tsx`. Those two **will compile-error** on the new value (the feature working — add a
`DISPUTED` entry to each). But OrderStatus is rendered/branched in **many other** files that do
**not** use `satisfies` (client dashboard `ui.tsx`, lab dashboard `page.tsx`, `order-detail`,
checkout, lab-fulfillment). Those will **silently miss `DISPUTED`** — no compile error, a latent
wrong/blank label in prod. Grep `grep -rln OrderStatus src/features src/app` and hand-audit every
badge/label map; per the standing compounding rule, migrate any non-`satisfies` map you touch to
`as const satisfies Record<OrderStatus,…>` so the next enum change errors instead of silently
drifting.

### Decision 2: exactly three new state-machine transitions — no others

In `src/domain/orders/state-machine.ts`:
- `COMPLETED → DISPUTED` (client initiates, within the dispute window)
- `DISPUTED → COMPLETED` (admin resolves in the lab's favour — payout releases)
- `DISPUTED → REFUND_PENDING` (admin resolves in the client's favour)

Every write to `Order.status` in this slice **must** call `isValidStatusTransition()` before the
write (existing invariant). Do not add convenience edges (e.g. `DISPUTED → CANCELLED`) — out of
scope unless the planner justifies one with a regulatory citation.

### Decision 3: the dispute window — configurable, and decide where it is enforced

Client may dispute a `COMPLETED` order within a window (e.g. 7 days from completion). **Decide:**
the window constant lives in the domain layer (not inline in the action), and the dispute action
throws / rejects if `now - order.completedAt > window`. State the value and its location in the
plan. An unhandled out-of-window case must surface (throw), not silently no-op.

### Decision 4 (the money risk): payout hold while DISPUTED — gate inside the settlement `$transaction`

A `DISPUTED` order must **block payout release**. The T-10 settlement handler
(`src/features/payments/payouts/`) moves `pendingBalance → availableBalance` and marks
`Payout.status = COMPLETED`. **Decide the guard:** the settlement write must not release funds for
an order whose status is `DISPUTED`. Use the established **`updateMany` with a guard predicate in
`where` + `count === 0` early-return** discipline (compare-and-set; never a bare `update`) — the
guard predicate includes the not-disputed condition. This is the canonical webhook
state-transition rule (see `payments/webhooks/handlers.ts` + `payouts/handlers.ts`). Getting this
wrong releases money during an open dispute — treat it as the highest-risk line in the ticket.

### Decision 5: resolution direction semantics — what `REFUND_PENDING` actually does here

`DISPUTED → REFUND_PENDING` sets the status; **decide whether T-19 also triggers an actual refund
or stops at the status transition** (refund execution likely stays a separate/manual flow — state
the boundary explicitly so the planner does not silently scope-creep a refund integration in).
`DISPUTED → COMPLETED` must un-hold the payout (the settlement guard in Decision 4 keys off
current status, so resolving back to COMPLETED naturally re-permits release on the next/again
delivery — confirm this interaction in the plan).

### Decision 6: two slices, two auth postures

- Client dispute initiation: `src/features/orders/dispute/` — the client must own the order
  (ownership guard, mirror `acceptQuote`/`rejectQuote` in `src/features/orders/order-detail/action.ts`
  — there is no `quote-respond` slice).
- Admin resolution: `src/features/orders/dispute-resolution/` — **two-layer ADMIN auth** (layout
  guard is layer 1; the Server Action re-checks `session.user.role === 'ADMIN'` independently,
  DL-001 TOCTOU). Both write transitions; both go through `isValidStatusTransition()`.

### Decision 7: ITA SLA documented; PII + no analytics

Document the ITA-compliant response-time SLA in the plan/README even if not code-enforced.
Dispute reason text is free-form user input — narrow `formData.get('reason')` with
`typeof === 'string'` before use (boundary-input rule); it may carry sensitive content, so apply
the same data-minimization posture and **add no analytics/event tracking** to these pages.

## Watch-points during implementation

- **Every `Order.status` write calls `isValidStatusTransition()`** before the write (Decision 2).
- **Payout-hold guard uses `updateMany` + guard predicate + `count === 0` early-return**, never a
  bare `update`, inside the settlement `$transaction` (Decision 4 — the money-risk line).
- **`findUnique` on `Order.@id`**; null guaranteed relation after an explicit `include` throws,
  only a genuinely missing order is `notFound()`.
- **`redirect()` after — never inside — any try/catch** in both Server Actions.
- **`formData.get('reason')` runtime-narrowed** (`typeof === 'string'`), never `as string`.
- **`DISPUTED` label coverage:** add it to the 2 `satisfies` maps (compile-error sites) AND
  hand-audit every non-`satisfies` OrderStatus badge/label map (client/lab dashboards,
  order-detail, checkout, lab-fulfillment) that won't error — migrate touched ones to `satisfies`.
- **No bare `toLocaleString()`** in any new client component rendering a dispute timestamp — render
  deterministically (fixed locale + `timeZone`) or format in the RSC (newest compounding rule).
- **Decimal/Date serialization** at any new RSC→client DTO boundary (`amount: string`, ISO dates).
- **Dispute window unhandled case throws**, does not silently no-op (Decision 3).
