# T-12 — Attachment uploads (client spec documents + lab result PDFs)

Plan document: `plans/T-12-attachment-uploads.md` (to be written)
Branch: `feat/T12-attachment-uploads`
Estimated sessions: 2–3 | Estimated wall time: ~3 hours
Why this is next: the last clean Phase-4 engineering ticket. T-13b is gated on a real ops
need; T-13c is deferred (privilege-escalation audit); T-19 (dispute/redress) is blocked on a
schema migration. T-12 completes the order artifact flow — a client attaches specification
documents to an order, and the lab attaches result PDFs at completion — closing the loop
between "order placed" and "results delivered."

T-12 is **substantially de-risked by T-15 + T-13 + T-18** — Cloudflare R2 is provisioned, the
presigned PUT (`generatePresignedPutUrl`) and on-demand presigned GET (`generatePresignedGetUrl`)
helpers exist in `src/lib/storage/r2.ts`, and the two-step upload + server-trusted-key pattern
is established across three merged slices. This is mostly *applying* that pattern to a second
model with two upload actors, hence the 2–3 session estimate.

**Key difference from T-15/T-18:** those used `LabDocument` (lab-scoped, has a
`status DocumentStatus` lifecycle, `r2Key @unique`). T-12 uses **`Attachment`** —
**order-scoped**, currently **unused in code**, with `fileUrl String` (not `r2Key`) and **no
status column**. The pattern transfers but the model shape forces three real decisions below.

## Pre-session — resolve these BEFORE calling the planner

Front-load these; do not let the planner re-open them mid-interview.

### Decision 1 (the big one): `Attachment.fileUrl` stores the SERVER-TRUSTED R2 OBJECT KEY — never a public/presigned URL

The `Attachment` model (schema line 253) predates the T-15 storage pattern and names the column
`fileUrl String`. **Do not persist a public or presigned URL in it.** Presigned URLs expire
(300 s) and a persisted public URL leaks the object indefinitely. T-13/T-18 store a
server-trusted **key** (`LabDocument.r2Key`) and mint a presigned GET **on demand** per
authorized access (`generatePresignedGetUrl(key)`, 300 s TTL). T-12 must do the same.

**DECIDE one of:**
- **(Recommended) Add `r2Key String @unique`** to `Attachment`, mirroring `LabDocument` exactly
  — gives uniqueness-based idempotency (re-confirm can't create a duplicate row) and makes the
  intent self-documenting. Small additive migration; `fileUrl` either drops or stays unused.
- Reuse `fileUrl` as the key column with a doc note (no migration, but `fileUrl` is a
  misleading name and gives no `@unique` idempotency guard).

Recommendation: **add `r2Key @unique`**. The idempotency guard alone justifies the migration,
and it keeps the two storage models congruent. State the choice in the plan.

### Decision 2: two actors, two permission guards, two size limits, possibly two MIME allowlists

`Attachment.attachmentType` is `AttachmentType` = `{ SPECIFICATION, RESULT, ACCREDITATION_CERTIFICATE }`.
T-12 wires **SPECIFICATION** and **RESULT** only — `ACCREDITATION_CERTIFICATE` is **dead** (T-18
used `LabDocument` instead; do not wire it).

| Actor | Type | Guard | Order status window | Size limit | MIME |
|-------|------|-------|---------------------|-----------|------|
| **CLIENT** | `SPECIFICATION` | `order.clientId === session.user.id` | early (at/after creation, pre-fulfilment) | 20 MB (`MAX_BYTES`) | pdf/jpeg/png |
| **LAB_ADMIN** | `RESULT` | `order.lab.ownerId === session.user.id` | at completion (IN_PROGRESS / COMPLETED) | **50 MB** (new `MAX_RESULT_BYTES`) | PDF-only? decide |

`src/lib/storage/constants.ts` currently exports only `MAX_BYTES = 20 MB`. Add
`MAX_RESULT_BYTES = 50 * 1024 * 1024` and pass the limit by attachment type through the
presign + validate path. Confirm whether `RESULT` is PDF-only (likely) vs the shared
pdf/jpeg/png allowlist; if different, add a per-type MIME allowlist (mirror the
`as const satisfies` allowlist style in `kyc-upload`/`accreditation-upload`).

### Decision 3: `Attachment` has NO status lifecycle — the row's existence IS the "uploaded" signal

`LabDocument` has `status PENDING→UPLOADED→VERIFIED/REJECTED` and a CAS confirm. `Attachment`
has **no status column**. Do not add one unless a review workflow is in scope (it is not —
attachments are not reviewed/approved). The flow is therefore **single-commit**:
1. `requestUploadUrl` action — validate type/size/MIME, ADMIN/CLIENT+ownership guard, return a
   presigned PUT URL for a freshly generated `r2Key` (do NOT create the row yet).
2. Client `PUT`s the bytes to R2.
3. `confirmUpload` action — create the `Attachment` row from the **server-trusted** `r2Key`
   (not client-supplied). `r2Key @unique` (Decision 1) makes a duplicate confirm a no-op /
   handled conflict rather than a second row.

There is no `updateMany` CAS here because creation is the commit; the concurrency guard is the
unique constraint, not a compare-and-set. (If Decision 1 reuses `fileUrl` with no `@unique`,
you LOSE this guard — another reason to add `r2Key @unique`.)

### Decision 4: where does the spec upload live — NOT inside the create-order `$transaction`

`create-order/action.ts` writes `Order` + `ClientProfile` in one `$transaction`. **Do not** fold
a file upload into it — the R2 PUT is a client-side async step and would either bloat the
transaction or couple order creation to upload success. **DECIDE: client attaches
SPECIFICATION on the order-detail page _after_ creation** (`src/features/orders/order-detail/`),
not as part of the create-order form. Order creation stays atomic; attachments are an
independent, idempotent follow-up. The lab attaches RESULT from the lab-fulfilment surface
(`src/features/orders/lab-fulfillment/`) at completion.

### Decision 5: rollout is additive — low risk

Unlike T-18, T-12 changes no gate and empties no surface. New columns + new actions + new UI on
existing pages. No behaviour-change rollout to communicate.

## Critical gotchas (carry into the plan)

1. **Store the server-trusted KEY, not a URL.** `Attachment.fileUrl`'s name is a trap — persist
   `r2Key` (add the column, `@unique`) and mint presigned GETs on demand (300 s), exactly like
   `LabDocument.r2Key` + `generatePresignedGetUrl`. Never persist a public/presigned URL.
2. **Two size limits.** SPECIFICATION 20 MB (`MAX_BYTES`), RESULT **50 MB** (`MAX_RESULT_BYTES`,
   new). Plumb the limit by `attachmentType`; do not hardcode 20 MB in the result path.
3. **`AttachmentType.ACCREDITATION_CERTIFICATE` is dead.** T-18 used `LabDocument`. Wire only
   SPECIFICATION and RESULT.
4. **Do not fold uploads into the create-order `$transaction`.** Spec upload happens on
   order-detail _after_ creation; result upload on lab-fulfilment at completion. Order creation
   stays atomic.
5. **The download guard is per-actor and security-critical.** A presigned GET hands out the
   bytes — re-check ownership (`order.clientId === userId` for the client; `order.lab.ownerId
   === userId` for the lab) inside the view action, server-side. UI hiding is not a control.
6. **Migrations are gitignored; dev DB is a `db push` workflow.** Apply the `r2Key` column with
   `npx prisma db push` (NOT `migrate dev`). `schema.prisma` is the committed source of truth.
   Carry a per-environment `db push` line into `docs/roadmap.md` (same as T-18's audit columns,
   which are also still owed per env).
7. **`vitest.unit.config.ts` include globs.** Every new `__tests__/` dir must be added to the
   `include` list or its tests silently never run (this bit T-18 — two files were missed until
   the glob was added).

## Implementation Discipline reminders (from CLAUDE.md)

- Action TOCTOU: re-check the actor's role + ownership inside every upload/confirm/view Server
  Action — page/layout guards do not protect actions.
- `findUnique` on `@unique` fields (`Order.id`, `Attachment.r2Key`) — never `findFirst`.
- After an explicit `include` (e.g. `order.lab`), a null relation → `throw`; missing row / wrong
  owner → `notFound()`.
- Boundary inputs runtime-narrowed (`typeof x === 'string'`); no `as string`; no masking
  `?? ''`/`?? null` on contract-guaranteed values. (T-18's one QR finding was exactly a missing
  `typeof` narrow on a `formData.get()` value — do not repeat it.)
- `redirect()` after — never inside — try/catch in Server Actions.
- RSC DTOs: `Date` → `.toISOString()`, `Decimal` → `.toFixed(2)` before crossing to a client
  component; DTO field types reflect the serialized form.
- Presigned GET key is server-trusted (`Attachment.r2Key`), never client-supplied; 300 s TTL.
- Per-type enum allowlists via `as const satisfies` — throw on an unknown type, never default.
